Choosing memory for enterprise agents.

Frontier ships as papers. Consumer ships as demos. Applied AI ships as systems, the discipline of taking a research result or a consumer-grade feature and converting it into an enterprise substrate with measured properties, named failure modes, and an adoption path. This entry closes the gap on one capability: long-term memory for coding-orchestration agents on an SDLC-history corpus.

01 Executive summary

Three findings, one decision.

Structured as a deployment-decision memo, not a benchmark recap. The three callouts below are the headline; the body sections that follow are the supporting argument.

Finding 1. Production-weighted retrieval separates the substrates by 36 points

The chunk-text retrieval approach (LightRAG) outperformed the entity-graph approach (Graphiti) by 36 percentage points when the test was weighted to match real production query frequency. Under naïve uniform weighting the two looked tied; the weighting decided the outcome. The bar is stated, the weights are stated, and the dispersion across executors is small enough to attribute the win to substrate architecture, not lucky pairings.

Finding 2. Reliability is binary across tiers

The top tier produced zero fabricated answers across 128 trials. The second tier produced four, across the same prompt set, the same eight tasks, the same executors. In enterprise terms this is the difference between a system one ships and a system one runs in shadow until further notice.

Finding 3. Bi-temporal metadata helps a smaller executor and hurts a more-capable one

The stronger executor sometimes re-quoted the older fact because the metadata told it the older fact existed. Context design must match executor capability; it is not a universal default until the formal closed-loop suite re-runs.

02 Why memory matters

Coding agents do not fail on the call. They fail across calls.

Single-turn coding tasks are largely solved by 2026: the strongest executor models close most well-specified one-shot prompts inside one or two iterations. The failures that matter for enterprise adoption appear in multi-turn workflows. The agent has to remember what was decided three turns ago, hold a project-wide invariant across files, and resist drifting into a different solution because the conversation has accumulated noise.

That capacity sits in the memory architecture, not the language model. The model produces a next token; the memory architecture decides which prior turns, files, and decisions are visible when that next token is produced. Enterprise teams do not buy "a model." They buy a system whose memory architecture decides whether a multi-day workflow holds together.

What "memory" means here

Three structural choices, in combination, define a memory architecture:

  • What is retained. Conversation turns, retrieved file chunks, summarised prior sessions, decision artefacts.
  • How retention is keyed. Recency window, semantic similarity, explicit tags, graph traversal, deterministic indexing.
  • What gets surfaced per turn. The slice of the retained set that the executor model actually sees on a given call.

The two architectures compared in this report, LightRAG (chunk-text) and Graphiti (entity-graph with bi-temporal metadata), differ on all three.

04 Test design

Two substrates, two executor tiers, eight tasks.

The matrix is small and deliberate: two memory architectures (LightRAG, Graphiti) paired with two executor capability tiers (frontier and economical), evaluated against eight repository-shaped task families, two passes per cell. 128 trials total. The eight task families are weighted by their production query frequency on the SDLC-history corpus; uniform weighting is reported alongside for comparison.

Hidden validators decide each task at the binary level: the agent either produced the right artefact or it did not. Refusal, when memory was designed to be insufficient, counts as a correct response in the reliability column.

The 36-point gap is the weighted score difference, not the raw correctness delta. Most of the spread is driven by recall and reliability on canonical-text queries. 06 Reliability

The fabrication test, and what passing it means.

Each trial included a designed insufficiency: a task where the substrate did not have the file or the prior decision required to answer. The correct behaviour is to refuse and ask. Fabrication is producing an answer that looks reasonable but cites the wrong document, the wrong supersession date, or invents an architecture invariant.

The result

LightRAG refused all 16 insufficiency cases (twice each across two passes) and produced zero fabrications across the 128 total trials. Graphiti fabricated on four cases, all in T2-executor pairings. The structural reason: when the entity graph carries a confidently-wrong supersession claim, the executor cannot disambiguate against the underlying source.

Reliability is not a smooth metric. A system that fabricates once per hundred is the same shape of system as one that fabricates four times per hundred; both require a human reviewer on every output. The difference between "passes the bar" and "fails the bar" is binary at zero.

09 Platform

What the substrate has to expose.

If the memory architecture is the thing that wins, the platform contract that wraps it needs four properties that today's gateways do not all have.

Per-session memory state, addressable

Each session needs an externalised memory state that survives executor model swap. If the executor changes mid-session, the memory layer cannot reset.

Audit of what was surfaced

For governance, the substrate has to log the slice of memory the executor actually saw on each call, not only the outgoing prompt. Otherwise an inspector cannot reconstruct why a particular answer was produced.

Refusal as a first-class outcome

"Refused: insufficient context" needs to be a recordable, queryable terminal state on the same axis as a successful response. Today most platforms treat it as a transport layer detail or drop it into freeform text.

Decoupled cost accounting

Memory operations (retrieval, summarisation, graph traversal) have their own cost shape. Cost-per-success only reads correctly when memory and executor billing are separable in the telemetry.

The DIAL ask

EPAM's substrate already exposes a translation adapter, a project-keyed gateway, and request-level telemetry. The four properties above are the next contract increment: memory state as a first-class side-channel, surfaced memory in the audit record, refusal as a terminal outcome, and decoupled cost accounting per memory operation.

10 Recommendations

Three for engineering, two for the platform.

  1. Standardise on LightRAG. Within EPAM, every coding-orchestration deployment that touches more than a single file moves to chunk-text retrieval by the end of Q3 2026.
  2. Cap Gemini Flash to scoped workflows. The cheaper executor passes the bar when the workflow is bounded. Publish the list of qualifying workflow shapes; reject pilots that fall outside it.
  3. Refuse-rate as an SLA. A coding-orchestration deployment publishes a refuse-rate alongside its success-rate. Customers see both; absence of refusals is a red flag, not a green one.
  4. Memory state as an exported artefact. The platform exposes the per-session memory state via a stable read API. Operators can inspect it; auditors can attach it to incident records; the next executor can read it on swap.
  5. Cost decomposition by memory op. The platform's telemetry separates memory operations from executor calls at the billing layer. Cost-per-success becomes legible across executor swaps.

11 Scope limits

What this report does not cover.

  • Non-coding workflows. The eight tasks are repository-shaped. A customer-support agent, a clinical-research agent, or a sales-ops agent has different memory pressure. Treat the findings as a strong prior, not a transfer.
  • Sessions over a month. The longest session in this matrix is five working days. Memory architectures have different decay profiles past that; the matrix does not reach there.
  • Multi-agent coordination. One executor, one memory, one session. Multi-agent setups introduce inter-agent memory which is its own pressure shape.
  • Adversarial input. The matrix is benign. Prompt-injection resilience of either architecture is the next report.

Methodology and raw artefacts. Adoption-bar weights, task families, retrieval traces, and evaluator prompts live alongside the report source on GitHub: agorokh/applied-ai-research. Re-running the matrix against either substrate is a clone-and-pytest exercise.

12 Closing

Applied AI is the work of moving the bar to the right place.

The headline number across this engagement is not the 36-point gap. It is that we know the gap because the bar was stated before the trials ran. Restate the bar, retest the architectures, republish the gap, that is the loop. This entry is one rotation of that loop, on one capability. The next rotation, on prompt-injection resilience, is in progress.
