01 Executive summary

Three findings, three parts.

The report is in three parts. Part I (§02) is the bridge: a small translation adapter that lets the Claude Code harness run against any upstream DIAL already brokers, on the same EPAM-compliant inference path every other LLM call already uses. Part II (§03) is the 192-trial matrix at AWS Bedrock list pricing. Part III (§04–§10) is the adapter-as-telemetry-plane reframe: the structured records the adapter writes on every request, the three latent tags that can be promoted without client changes, and the business-metadata joins that would unlock per-team, per-project, per-industry reporting in EPAM's production environment.

Finding 1. The bridge runs on the EPAM-compliant inference path

Claude Code 2.1 talks to any DIAL-hosted upstream through one Anthropic-to-OpenAI translation adapter. The harness routes traffic through <dial-gateway-host>, the same gateway EPAM already uses for sanctioned LLM access, instead of calling api.anthropic.com directly. 192 trials produced zero authentication incidents and required no Claude Code source modifications. From DIAL's perspective the requests look like any other OpenAI-shape call; the adapter handles the Anthropic-shape input/output on the client side.

Finding 2. Qwen3-Coder 480B is the cheapest 100%-pass model at $0.101 per success

At AWS Bedrock list pricing verified 2026-05-14 (US East Ohio), Qwen3-Coder 480B is the cost leader of the 100%-pass cluster: 38% below Claude Haiku 4.5, 4.7× below Sonnet 4.6, 6.8× below Opus 4.7. Below the Sonnet tier, three OSS upstreams (Mistral Devstral 123B, Qwen3-Coder 30B-A3B, DeepSeek v3.2) fail the harness with a scale-driven, vendor-independent signature. The ranking inside the 100%-pass cluster (Qwen 480B < Haiku 4.5 < Sonnet 4.6 < Opus 4.7 on $/success) is stable under markup factors below ~6.7× on Qwen input rates; EPAM DIAL contract pricing may shift absolute magnitudes but does not flip the ranking inside that range.

Finding 3. The adapter retains structured per-request telemetry today

On every response_out the adapter writes a structured JSON log record (token counts, tool inventory hash, MCP servers seen, cache strategy, elapsed milliseconds, Bedrock-verified cost_usd_estimate) to InfluxDB measurements adapter_metrics, claude_code_events, and claude_code_traces. Today those records carry two business-relevant dimensions (client_name, target_model_family); three identifiers already in the wire (session.id, tool_inventory_hash, mcp_servers_seen[]) can be promoted to tags with no client change. Business-context tags (project, team, industry, use-case) are not wired in this POC; centralising the adapter inside DIAL puts EPAM in a position to supply those joins from existing project metadata, subject to data-governance policy.

For platform teams: the bridge validates the routing pattern; the matrix validates the economics; the telemetry plane is the part that converts the POC from a per-laptop convenience into a centralised observation point. §10 carries the productionisation proposal: port the translation core into DIAL's ingress codepath, behind a feature flag, with 2–3 friendly pilot projects and a 2–4 week observation window.

02 Part I · The bridge

One translation adapter, eight upstreams, the same EPAM gateway.

The adapter accepts POST /v1/messages (the request shape Claude Code emits), translates it to DIAL's OpenAI-style chat completions, and streams the SSE response back in the shape Claude Code expects. A thin wrapper around the claude CLI sets the harness's standard environment variables to route traffic at the adapter and to redirect Claude Code's built-in /model picker (Default, Sonnet, Haiku) at any DIAL deployment without restarting the session.

Compliance posture

The bridge routes Claude Code through <dial-gateway-host> using a DIAL project key the engineer already has under their EPAM enterprise subscription. Traffic that would otherwise leave the EPAM perimeter for api.anthropic.com stays on the sanctioned inference path. The engineer's side of the change is one environment variable (ANTHROPIC_BASE_URL); the gateway side is unchanged in this POC.

Two technical realities to flag for Part III

First, the adapter strips cache_control on every non-Anthropic upstream (else DIAL returns 502 No route), a known gateway gap. Second, on every response_out the adapter emits a structured GFLog record with token counts, tool inventory hash, MCP servers seen, cache strategy, elapsed milliseconds, and a Bedrock-verified cost_usd_estimate field. Today that record lands in Influx adapter_metrics as a tag-and-field rollup, and Grafana shows it as KPIs. With business-identity headers forwarded, the same record carries project, team, and industry. That is what Part III addresses.

For platform teams: the adapter is the only component in the request path that sees the full Anthropic-shape request, the translated upstream call, and the full streamed response. That position is what makes it a natural telemetry capture point, regardless of where it ends up running. Centralising it into DIAL preserves that property and adds the per-tenant identifiers the wire already needs.

04 Part III · Data plane

The adapter is the only component that sees the full request.

On every response_out the adapter writes a structured JSON log record summarising token counts, tool inventory, cache-control behaviour, elapsed latency, the upstream stop_reason, and a Bedrock-priced cost_usd_estimate derived from the operator price table. Two sidecar exporters tail those records into InfluxDB measurements (adapter_metrics, claude_code_events) which Grafana queries to produce the operational dashboard.

Today those records are tagged by client_name and target_model_family. That is sufficient for operational health views ("how is the wire doing?") but not for per-session, per-project, or per-team analysis. §05 separates what is captured today, what is in the wire but not yet promoted to Influx tags, and what would require business-identity headers Claude Code is not sending today.

The strategic reframe: a translation adapter is not just a wire converter. It is the structured observation point for the AI-native SDLC, a position no other component in the stack occupies. Centralising it into DIAL preserves that observation point and adds the per-tenant identifiers EPAM's existing project metadata can supply, subject to data-governance policy.

05 Telemetry layers

Three layers: captured today, in the wire, requires headers.

To avoid overstating the POC's data plane, this section separates three distinct layers: (A) fields that already reach InfluxDB and surface in Grafana today, (B) fields the adapter captures in its JSON log but does not currently promote to an Influx tag (mechanical to wire, no client change), and (C) business-context fields that would require headers Claude Code is not sending today. The POC implements Layer A. Layer B is mechanical follow-up. Layer C is the architectural ask in §10.

07 Director questions

Five management-level questions the joins would unlock.

Each row pairs a real management-level question with the data join it would require. Each join is "Layer-A telemetry already in InfluxDB + one Layer-C business-metadata source." None require new collection from the wire; only the connection to business metadata that an enterprise environment is positioned to supply, subject to data-governance policy. The five questions below are illustrative for any delivery firm running a coding-agent gateway; substitute your own routing policy, contract structure, and team identity for the specifics.

#The questionThe data join
Q1 Which delivery teams get real productivity lift from AI vs just burning tokens? team_identity + task_completion_rate + tokens_per_session + Jira velocity delta. Surfaces teams with high token volume but low completion, prompt-discipline problems, not tool problems.
Q2 What is our true AI cost per billable hour delivered, by account? project_code + contract_phase + blended cost_per_request + timesheet. Converts flat AI infra cost into a per-account P&L input.
Q3 Is our OSS-first routing policy holding up under real conditions, or are engineers bypassing it? client_variant + model_family + user_identity. Detects shadow usage of Anthropic endpoints when DIAL routes to OSS, preference vs policy gap.
Q4 For which use-case archetypes should we build fine-tuned OSS alternatives, and where is proprietary still necessary? use_case_label + pass_rate + model_family + cost_per_task. Produces a capability gap matrix: OSS-sufficient vs proprietary-justified, by task type.
Q5 Can we commit to a client that their AI-assisted delivery stays within a defined cost envelope? project_code + token_volume_trend + model_routing_policy + contract value. Enables AI cost SLAs in SOWs, a competitive differentiator in fixed-price engagements.

08 Use cases

Three Layer-C joins, sketched, not measured.

The three scenarios below are illustrative. None of the numbers come from this POC's data; the POC's wire does not yet carry the business identifiers each scenario needs (Layer C in §05). They are included to show the form of the analyses that become possible if the join is wired. Replace numbers with real measurements once the join lands.

Use case 1 · Per-industry routing decision

Hypothetical scenario. Suppose Claude Code requests carried an industry_vertical tag (mechanism out of scope here). Pharma-tagged repositories with paths under src/<regulated>/ would be expected to exercise long-context comprehension under regulation; non-regulated paths in the same monorepo would not. If the joined telemetry showed a meaningful per-path acceptance-rate gap between Anthropic Sonnet and Qwen3-Coder 480B on the regulated paths, that would justify a per-path routing rule, not a per-tenant override. Whether such a gap exists is what the joined data would tell us.

Use case 2 · Per-team cost reporting

Hypothetical scenario. Suppose two adjacent EPAM accounts ran Claude Code through the same DIAL instance with team_id on each request. Account A's data-eng team and Account B's frontend team almost certainly have different per-session token profiles (depth of tool_result chains, cache-read ratios, average turn count). Slicing the existing Layer-A measurements by team_id would produce two distinct cost-per-session curves and, with an accept-signal joined, two distinct cost-per-merged-PR curves. The recommended Sonnet-tier model for each team would emerge from data, not from a guess.

Use case 3 · Use-case-driven adaptive routing

Hypothetical scenario. Suppose use_case_label were derived per request from commit-message + diff shape (e.g. refactor-multi-file, trap-comprehension, spec-trace). After enough sessions, the Layer-A telemetry sliced by use_case_label would let the platform team test whether some task classes are cheaper to serve on Qwen 480B at unchanged acceptance rates while others require an Anthropic tier. The output would be a small YAML routing rule per task class, with the evidence (sample size, observed success delta, cost delta) attached. Whether the gap is large enough in any one task class to justify a rule is an empirical question.

Common dependency across all three

The three scenarios share a single enabling step: business-identity headers on each Claude Code request (Layer C in §05), reaching the existing Influx measurements as tags. The downstream queries (per-industry path-level acceptance, per-team cost curves, per-task-class routing rule) are Flux against measurements the adapter already populates. None of the queries are runnable on POC data today; their structural form is what §08 illustrates.

09 Applications

Three application classes that follow from the same join.

Three application classes that follow directly from the data the adapter already emits, once business-metadata headers are forwarded and the three latent tags are promoted. Each is a concrete reporting or routing capability, not a new product proposal.

Application A · Per-account AI consumption reporting

The same Grafana panels that today show cost-by-model can be pivoted by project_code and account_id to produce per-account spend reports. Combined with timesheet data (or any project-level effort signal), this yields a per-engagement cost-per-engineer-hour figure that is currently aggregated implicitly across the entire DIAL deployment. The reporting surface is a re-query of existing measurements; no additional collection is required.

Application B · Segment-specific model fitness reports

The 192-trial evaluation methodology in Part II is reusable. With industry_vertical and use_case_label tags promoted, the same matrix can be sliced per segment to produce evidence-based fitness statements (for instance, "on FinServ refactoring tasks, Qwen 480B matches Claude Sonnet's pass rate at a 4.7× cost advantage on this task set"), accompanied by the underlying transcripts and validators. The output is a research note, not a marketing claim.

Application C · Pattern library indexed by observed efficiency

Tool-call sequences are already captured at wire level (tool_calls_by_name, tool_inventory_hash, mcp_servers_seen[]). Once joined to use_case_label and an accept-signal, sequences with low loop depth, high completion rate, and low token cost can be identified, documented, and offered as reusable prompt or agent templates. This is an empirical extraction from telemetry, not a designed library.

What ties Applications A–C together

All three depend on the same enabling work in §10: business-identity header forwarding and three tag promotions in the exporter. None require new collection, retraining, or SDK changes. Each yields a reporting or routing capability whose evidence base is the transcripts already on disk.

10 Productionisation

From a per-laptop adapter to a centralised gateway feature.

This section is the productionisation pattern any organisation running a sanctioned-inference gateway can apply to its own deployment; the DIAL-specific instantiation below is what one EPAM engagement chose. The pattern: take the validated POC adapter (an Anthropic-shape ingress that translates to OpenAI-shape outbound) and move it inside the gateway itself. Once that exists, any developer holding an enterprise gateway license can point Claude Code at the gateway without running a local adapter, and project-level configuration on the gateway side decides which upstream their picker slots resolve to.

State today: POC running on the engineer's laptop

The POC runs the adapter locally and a thin wrapper around the claude CLI points it at the loopback port. The adapter authenticates to DIAL with the engineer's enterprise project key, translates the wire format, and forwards. Telemetry lands locally in InfluxDB and Grafana. This is the shape that produced the 192 evaluation trials in Part II. It is the right shape to validate the approach; it is not the right shape to roll out across delivery teams.

Proposed end state · the translator inside DIAL

DIAL gains an Anthropic-compatible ingress (e.g. POST /anthropic/v1/messages) that internally executes the same translation the POC adapter performs today and routes to the same Bedrock-backed deployments DIAL already brokers. From the developer's side, the change collapses to one configuration line in their Claude Code environment:

# before: direct Anthropic
ANTHROPIC_BASE_URL=https://api.anthropic.com

# after: EPAM-compliant inference path via DIAL
ANTHROPIC_BASE_URL=https://<dial-gateway-host>

No new credentials (DIAL project key already in the engineer's enterprise subscription). No local processes. No new auth model. The same EPAM data-governance, audit, and rate-limit posture that already applies to every DIAL call applies to Claude Code calls.

Pilot rollout proposal

  1. Build the in-DIAL adapter. Port the translation core (anthropic_to_openai / openai_to_anthropic_response / SSE bridge in app.py) into DIAL's ingress codepath, behind a feature flag.
  2. Select 2–3 friendly pilot projects. Delivery teams already using Claude Code informally, ideally with diverse stacks (one heavy-Python, one TypeScript, one mixed). Centrally update their DIAL project configuration to enable the Anthropic-compatible ingress.
  3. Observe for 2–4 weeks. Existing DIAL telemetry surfaces per-project token / cost / latency; Claude Code-specific signals continue to land in the Layer-A measurements described in §05.
  4. If telemetry holds and cost-per-success matches the Part II numbers, expand. Add cohorts of additional projects in waves, each wave with a brief feedback loop to capture friction before the next cohort.
  5. Wire Layer C alongside the rollout. The same pilot is the natural moment to add whichever business identifiers EPAM's data-governance policy permits (e.g. project_code tag) so the reports in §07–§09 can be populated from real data instead of placeholders.

What the proposal does not require

  • No model retraining and no Anthropic vendor relationship change; everything is brokered through existing DIAL deployments.
  • No new auth mechanism for developers; DIAL project keys already cover Claude Code traffic once it is routed through the gateway.
  • No SDK changes to Claude Code itself.
  • No new local processes on developer machines after the POC stage ends.

Engineering scope estimate

The translation core in the POC adapter is around 1500 lines of Python concentrated in two functions plus the SSE streaming bridge. Porting into DIAL's ingress codepath is a translation rewrite, not a new design; the wire-shape conversions are stable. The team-internal effort is, to first order, the time to integrate the existing translation logic with DIAL's existing authentication, project routing, and observability paths, not the time to design from scratch. The POC's test suite (in the adapter's test suite) carries forward as the conformance bar.

11 Caveats

What this document does not claim.

  • Single-operator data. All 192 trials originate from one engineer's laptop, one DIAL deployment, one network path to <dial-gateway-host>. Latency tails include network RTT variance that production deployments would see differently.
  • Synthetic task surface. The 8-task portfolio is reproducible and discriminating but is not a stand-in for any specific real codebase. T8 (refactor → test → document) is the closest to a realistic refactor; T1–T7 are unit-test shaped.
  • Bedrock list pricing, not EPAM DIAL contract pricing. Every $/M rate in the cost table is taken from the AWS Bedrock pricing console (US East Ohio, 2026-05-14). EPAM DIAL contract pricing may include a markup or discount that shifts absolute magnitudes; the qualitative ranking inside the 100%-pass cluster is stable under markup factors below ~6.7× on Qwen 480B input rates.
  • Part III is a description of latent capability, not a deployed system. The three tag promotions and header allowlist described in §10 are not in production. Illustrative numbers in §06, §07, and §08 are not measurements from current data and are labelled as such.
  • No human-eval cross-check. Hidden validators catch syntactic / functional correctness but not code quality. A model could pass T2 by producing semantically-correct-but-stylistically-poor code; the matrix would call this a pass.

12 Closing

Bridge, matrix, telemetry plane. Three parts, one rollout.

Part I shows the bridge functions end-to-end on this stack. Part II reports a single-machine 192-trial benchmark with a clear ranking under Bedrock list pricing: Qwen3-Coder 480B is the cheapest 100%-pass model on this task surface; the Haiku tier remains Anthropic-only at the observed scales; three OSS upstreams fail the harness with a shared scale-driven signature. Part III inventories the per-request telemetry the adapter already emits and identifies a small, scoped change (~50 LoC across the adapter and one exporter) that would make those records joinable with project, team, industry, and use-case metadata.

None of this is a vendor benchmark or a commercial statement. It is an internal description of one stack's observed behaviour and a list of next steps the same stack can support, written so colleagues can reproduce or extend any number in it. The strategic angle is the telemetry plane: centralising the adapter inside DIAL preserves the structured observation point and adds the per-tenant identifiers EPAM's existing project metadata can supply. The rollout follows the §10 proposal; the data join follows EPAM's data-governance policy.

Applied AI as a practice is the work of taking a frontier or consumer-grade capability and converting it into an enterprise substrate with measured properties, named failure modes, and an adoption path. This report is one instance of that work, on one capability: a coding-orchestration harness on the EPAM-compliant inference path, with the telemetry plane that the centralised version would unlock.
