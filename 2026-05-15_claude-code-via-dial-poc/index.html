00 Cover · Applied AI · The middle band

Frontier ships coding agents as products.
Applied AI turns them into governed systems.

The AI market splits three ways in 2026: research that proves a capability, consumer products that dramatise it, and enterprise systems that have to survive scale, tenancy, audit, and the governance contract every regulated delivery context insists on. Applied AI is the middle band — the discipline of taking a research-grade or consumer-grade capability and converting it into an enterprise substrate with measured properties, named failure modes, and an adoption path. This entry closes the gap on one capability: routing Anthropic's Claude Code 2.1 through EPAM's DIAL gateway, measuring eight upstreams under one fixture, and treating the adapter as a structured observation point for the AI-native SDLC.

01 Executive summary

Three findings, one strategic angle.

Frontier labs ship coding harnesses (Claude Code, Codex CLI, Cursor) that point at the vendor's own endpoint by default. Consumer products lock the same harness to a single upstream. Neither layer answers the enterprise question: can the harness run on the gateway every LLM call already goes through, on whichever upstream the cost-quality curve actually favours, with the telemetry an enterprise needs to operate it? The three findings below answer yes — and surface the strategic angle: the wire is already a structured observation point.

Scope: eight upstreams measured through DIAL on one harness; Cursor, Aider, Codex CLI, OpenHands, and the various Continue / Cline stacks sit outside. The pattern transfers; the magnitudes do not transfer without re-measurement on another stack.

Finding 1 — The bridge works on the EPAM-compliant inference path

Claude Code 2.1 talks to any DIAL-hosted upstream through one Anthropic-to-OpenAI translation adapter. 192 trials produced zero authentication incidents and required no Claude Code source modifications.

Finding 2 — Cost-per-success at Bedrock list pricing

Qwen3-Coder 480B is the cheapest 100%-pass model at $0.101 per successful task — 38% below Claude Haiku 4.5, 4.7× below Sonnet 4.6, 6.8× below Opus 4.7. Three OSS upstreams below the Sonnet tier fail with a scale-driven, vendor-independent signature.

Finding 3 — Per-request telemetry, wired today plus a clean follow-up

Each request writes a structured record with tokens, tool inventory, cache behaviour, latency, and Bedrock-priced cost. Two business dimensions (client, model family) are wired today. Three more identifiers already in the wire can become tags with no client change. Business-context tags — project, team, industry, use-case — remain a follow-up, not a current capability.

02 Part I · The bridge and the EPAM-compliant path

One translation adapter; eight upstreams; the same EPAM gateway every other LLM call already goes through.

The adapter accepts POST /v1/messages (the request shape Claude Code emits), translates it to DIAL’s OpenAI-style chat-completions, and streams the SSE response back in the shape Claude Code expects. A thin wrapper around the claude CLI sets the harness’s standard environment variables to route traffic at the adapter and to redirect Claude Code’s built-in /model picker (Default, Sonnet, Haiku) at any DIAL deployment without restarting the session.

Compliance posture. The bridge routes Claude Code through ai-proxy.lab.epam.com using a DIAL project key the engineer already has under their EPAM enterprise subscription. Traffic that would otherwise leave the EPAM perimeter for api.anthropic.com stays on the sanctioned inference path. From DIAL’s perspective these requests look the same as any other OpenAI-shape call (same project key, same project routing, same logs) with the adapter handling Anthropic-shape input/output on the client side. The engineer’s side of the change is one environment variable (ANTHROPIC_BASE_URL); the gateway side is unchanged in this POC.

DIAGRAM · 02 · Bridge topology · eight upstreams behind one wire

Two technical realities to flag here for Part III. First, the adapter strips cache_control on every non-Anthropic upstream (else DIAL returns 502 No route), a known gateway gap. Second, on every response_out the adapter emits a structured GFLog record with token counts, tool inventory hash, MCP servers seen, cache strategy, elapsed milliseconds, and a Bedrock-verified cost_usd_estimate field. Today that record lands in Influx adapter_metrics as a tag-and-field rollup, and Grafana shows it as KPIs. Tomorrow, with one extra header, the same record carries project, team, and industry. That is what Part III addresses.

The pre-measurement prior was wrong

The pre-measurement prior expected the Sonnet-tier OSS swap to be viable at higher cost, the implicit posture in most Q1 2026 industry write-ups. The verified Bedrock-list-pricing pass reversed that call within the same evaluation window: Qwen3-Coder 480B is the economic optimum on this matrix at identical 100% pass on the eight tasks, not a fallback for cost-sensitive teams. The retraction is itself a finding: a benchmark that updates a prior belief is worth more than one that confirms the received wisdom.

Where this sits in the public landscape

Prior public work on coding-agent benchmarking is rich on the model side (the SWE-bench-Verified leaderboard, the Aider polyglot benchmark, BigCodeBench, the published vendor numbers for Claude Code on internal Anthropic benchmarks). It is sparse on the harness-routed-through-an-enterprise-gateway side. To the best of public-source review, this is the first non-vendor evaluation that runs Claude Code 2.1 specifically as the harness against the OSS-coder cohort (Qwen, Kimi, MiniMax, DeepSeek) on AWS Bedrock list pricing under one shared fixture. The published OSS coder leaderboards converge on similar capability rankings inside the 100%-pass cluster; what this report contributes is the gateway-routed economics. If counter-examples exist, the report is happy to be the second.

04 Part III · The adapter as a telemetry plane

What the adapter records on every request.

The adapter is the only component in the request path that sees the full Anthropic-shape request, the translated upstream call, and the full streamed response. On every response_out it writes a structured JSON log record summarising token counts, tool inventory, cache-control behaviour, elapsed latency, the upstream stop_reason, and a Bedrock-priced cost_usd_estimate derived from the configured price table. Two sidecar exporters tail those records into InfluxDB measurements (adapter_metrics, claude_code_events) which Grafana queries to produce the operational dashboard.

Today those records are tagged by client_name and target_model_family. That is sufficient for operational health views (“how is the wire doing?”) but not for per-session, per-project, or per-team analysis. Sections 05-08 walk through which fields are already captured, which are present in the wire but not yet promoted to Influx tags, and what kinds of joined queries become available once those three tag promotions land.

08 Part III · Three illustrative use cases

What the layer-C join enables, sketched, not measured.

The three scenarios below are illustrative. None of the numbers in them come from this POC’s data, the POC’s wire does not yet carry the business identifiers each scenario needs (see Layer C in Section 05). They are included to show the form of the analyses that become possible if the layer-C join is wired, and to give the platform team something concrete to react to. Replace numbers with real measurements once the join lands.

Use case 1 · Per-industry routing decision · illustrative

Hypothetical scenario. Suppose Claude Code requests carried an industry_vertical tag (mechanism out of scope here). Pharma-tagged repositories with paths under src/(gxp|fda)/ would be expected to exercise long-context comprehension under regulation; non-regulated paths in the same monorepo would not. If the joined telemetry showed a meaningful per-path acceptance-rate gap between Anthropic Sonnet and Qwen-Coder 480B on the regulated paths, that would justify a per-path routing rule, not a per-tenant override. Whether such a gap exists is what the joined data would tell us.

Use case 2 · Per-team cost reporting · illustrative

Hypothetical scenario. Suppose two adjacent EPAM accounts ran Claude Code through the same DIAL instance with team_id on each request. Account A’s data-eng team and Account B’s frontend team almost certainly have different per-session token profiles (depth of tool_result chains, cache-read ratios, average turn count). Slicing the existing layer-A measurements by team_id would produce two distinct cost-per-session curves and, with an accept-signal joined, two distinct cost-per-merged-PR curves. The recommended Sonnet-tier model for each team would emerge from data, not from a guess.

Use case 3 · Use-case-driven adaptive routing · illustrative

Hypothetical scenario. Suppose use_case_label were derived per request from commit-message + diff shape (e.g. refactor-multi-file, trap-comprehension, spec-trace). After enough sessions, the layer-A telemetry sliced by use_case_label would let the platform team test whether some task classes are cheaper to serve on Qwen 480B at unchanged acceptance rates while others require an Anthropic tier. The output would be a small YAML routing rule per task class, with the evidence (sample size, observed success delta, cost delta) attached. Whether the gap is large enough in any one task class to justify a rule is, again, an empirical question.

Common dependency across all three

The three scenarios share a single enabling step: business-identity headers on each Claude Code request (Layer C in Section 05), reaching the existing Influx measurements as tags. The downstream queries (per-industry path-level acceptance, per-team cost curves, per-task-class routing rule) are Flux against measurements the adapter already populates. None of the queries are runnable on POC data today; their structural form is what Sections 06-08 illustrate.

11 Part IV · Applications enabled by the enriched plane

What becomes possible once the joins are wired.

Three application classes that follow directly from the data the adapter already emits, once business-metadata headers are forwarded and the three latent tags are promoted. Each is a concrete reporting or routing capability, not a new product proposal.

Application A · Per-account AI consumption reporting

The same Grafana panels that today show cost-by-model can be pivoted by project_code and account_id to produce per-account spend reports. Combined with timesheet data (or any project-level effort signal), this yields a per-engagement cost-per-engineer-hour figure that is currently aggregated implicitly across the entire DIAL deployment. The reporting surface is a re-query of existing measurements; no additional collection is required.

Application B · Segment-specific model fitness reports

The 192-trial evaluation methodology in Part II is reusable. With industry_vertical and use_case_label tags promoted, the same matrix can be sliced per segment to produce evidence-based fitness statements (for instance, “on FinServ refactoring tasks, Qwen 480B matches Claude Sonnet’s pass rate at a 4.7× cost advantage on this task set”), accompanied by the underlying transcripts and validators. The output is a research note, not a marketing claim.

Application C · Pattern library indexed by observed efficiency

Tool-call sequences are already captured at wire level (tool_calls_by_name, tool_inventory_hash, mcp_servers_seen[]). Once joined to use_case_label and an accept-signal, sequences with low loop depth, high completion rate, and low token cost can be identified, documented, and offered as reusable prompt or agent templates. This is an empirical extraction from telemetry, not a designed library.

What ties Applications A-C together

All three depend on the same enabling work in Section 12: business-identity header forwarding and three tag promotions in the exporter. None require new collection, retraining, or SDK changes. Each yields a reporting or routing capability whose evidence base is the transcripts already on disk.

12 Part IV · Productionisation proposal

From a per-laptop POC adapter to a centralised feature inside DIAL.

The proposal this report supports is to take the validated POC pattern (an Anthropic-shape ingress that translates to DIAL’s OpenAI-style outbound) and move it inside the AI proxy itself. Once that exists, any EPAM developer holding an enterprise DIAL license can point Claude Code at ai-proxy.lab.epam.com without running a local adapter, and project-level configuration on the DIAL side decides which upstream their picker slots resolve to.

State today: POC running on the engineer’s laptop

The POC runs the adapter locally and a thin wrapper around the claude CLI points it at the loopback port. The adapter authenticates to DIAL with the engineer’s enterprise project key, translates the wire format, and forwards. Telemetry lands locally in InfluxDB and Grafana. This is the shape that produced the 192 evaluation trials in Part II. It is the right shape to validate the approach; it is not the right shape to roll out across delivery teams.

Proposed end state · the translator inside DIAL

DIAL gains an Anthropic-compatible ingress (e.g. POST /anthropic/v1/messages) that internally executes the same translation the POC adapter performs today and routes to the same Bedrock-backed deployments DIAL already brokers. From the developer’s side, the change collapses to one configuration line in their Claude Code environment: 

DEVELOPER ENVIRONMENT# before: direct Anthropic
ANTHROPIC_BASE_URL=https://api.anthropic.com

# after: EPAM-compliant inference path via DIAL
ANTHROPIC_BASE_URL=https://ai-proxy.lab.epam.com

No new credentials (DIAL project key already in their enterprise subscription). No local processes. No new auth model. The same EPAM data-governance, audit, and rate-limit posture that already applies to every DIAL call applies to Claude Code calls.

The actual request · what changes for the developer

Engineers continue to install and run Claude Code the same way. The picker still shows Default / Sonnet / Haiku. The difference is administrative: a per-project DIAL configuration decides which upstream each picker slot resolves to, exactly the way DIAL today decides which OpenAI deployment id a project key sees. Some projects might keep all three slots on Anthropic Bedrock; others might map the Sonnet slot to Qwen 480B based on the cost/quality evidence in Part II.

Pilot rollout proposal

  1. Build the in-DIAL adapter. Port the translation core (anthropic_to_openai / openai_to_anthropic_response / SSE bridge in app.py) into DIAL’s ingress codepath, behind a feature flag.
  2. Select 2-3 friendly pilot projects. Delivery teams already using Claude Code informally, ideally with diverse stacks (one with heavy Python, one with TypeScript, one with mixed). Centrally update their DIAL project configuration to enable the Anthropic-compatible ingress.
  3. Observe for 2-4 weeks. Existing DIAL telemetry surfaces per-project token / cost / latency; Claude Code-specific signals continue to land in the layer-A measurements described in Section 05.
  4. If telemetry holds and cost-per-success matches the Part II numbers, expand. Add cohorts of additional projects in waves, each wave with a brief feedback loop to capture friction before the next cohort.
  5. Wire layer-C alongside the rollout. The same pilot is the natural moment to add whichever business identifiers EPAM’s data-governance policy permits (e.g. project_code tag) so the reports in Sections 06-11 can be populated from real data instead of placeholders.

What the proposal does not require

  • No model retraining and no Anthropic vendor relationship change, everything is brokered through existing DIAL deployments.
  • No new auth mechanism for developers, DIAL project keys already cover Claude Code traffic once it’s routed through the gateway.
  • No SDK changes to Claude Code itself.
  • No new local processes on developer machines after the POC stage ends.

Engineering scope estimate

The translation core is around 1500 lines of Python concentrated in two functions plus the SSE streaming bridge. Porting into DIAL’s ingress codepath is a translation rewrite, not a new design; the wire-shape conversions are stable. The engineering effort is, to first order, the time to integrate the existing translation logic with DIAL’s existing authentication, project routing, and observability paths, not the time to design from scratch. The adapter’s test suite (52 tests) carries forward as the conformance bar.

The sanitised public adapter

Iteration 2 of this report ships alongside a sanitised public export of the adapter at github.com/agorokh/sdlc-dial-adapter. The public export preserves the adapter's evaluated core: the translation logic in app.py, the Dockerfile, the per-request JSON log schema (observability/EVENT_SCHEMA.md), the Grafana overview dashboard, and the 52-test conformance suite. Anyone with a DIAL key can clone it and reproduce the §03 matrix against the same upstreams. What remains private is deployment-specific scaffolding (the price-table calibration against EPAM contract pricing, the dashboard presets tied to a specific Influx tenant, the launcher tooling tied to a single workstation) - not the evaluated mechanism itself. Per the repo's own framing: "Part of EPAM's DIAL ecosystem of SDLC productivity experiments. This repo is one of the export paths from a larger internal sandbox: when an experiment proves it should be reusable, we extract the minimum subset needed to run it standalone and publish here."

A reader who wants to run the experiment themselves can clone the public repo and follow PORTABILITY.md; a reader who wants to read the wire-level findings in code form can read the classifier / model-decoupling finding and the Bedrock OSS hardening finding that this report’s §05-§07 are condensed from.

Iteration-2 substrate update · Kimi K2.5 as the second small model alongside Qwen

Iteration 1 of this report (2026-05-15) treated each OSS upstream as a single substitution candidate for the main loop. Subsequent work on the adapter surfaced a more useful pattern that has to be stated plainly before the configuration makes sense: Claude Code 2.1 runs three model paths internally - the main coding loop, an auto-mode safety classifier, and a background namer / summariser - and only one of those paths (the summariser) is rerouteable from a client environment variable today. The classifier path is hardcoded to the main-loop model in Claude Code 2.1.x; the summariser path follows ANTHROPIC_SMALL_FAST_MODEL. That distinction is the point of this update: the cost lever lives on summariser calls only, not on the whole agent.

The adapter runs with three model slots wired separately, not one: 

CLIENT ENV · MULTI-MODEL DECOUPLING# Main loop · capable Sonnet-tier OSS
ANTHROPIC_DEFAULT_OPUS_MODEL="qwen.qwen3-coder-480b-a35b-v1:0"

# Sonnet slot · second capable OSS model running alongside
ANTHROPIC_DEFAULT_SONNET_MODEL="moonshotai.kimi-k2.5"

# Haiku slot
ANTHROPIC_DEFAULT_HAIKU_MODEL="minimax.minimax-m2.5"

# Decouples background namer / summariser onto a cheaper slot
# so it does not burn the main loop's minute-token budget.
ANTHROPIC_SMALL_FAST_MODEL="moonshotai.kimi-k2.5"

The decoupling is a real cost lever. Background namer and summariser calls fire at high frequency relative to the main loop; routing them to a cheaper second model keeps the main loop on the capable tier and the bulk of cheap calls off it. I verified the split with the adapter's request_in log: across one representative session the secondary model handled 17 namer / summariser calls in parallel with 17 main-loop calls on Qwen 480B. The 17:17 ratio confirmed that both summary-related call paths were rerouted, not only one of them - the classifier-still-on-main-loop limit means the verification has to be log-grep, not a single counter. Both finding details are in the 2026-05-15 classifier / decoupling finding.

What the decoupling does not buy you

ANTHROPIC_SMALL_FAST_MODEL decouples the namer / summariser path only. The auto-mode safety classifier path is hardcoded to the main-loop model in Claude Code 2.1.x and ignores the env var. A controlled experiment confirmed this: the classifier_request log always reports the main-loop model regardless of the small-fast setting. Practical workaround for hot classifier load is permissions.allow in .claude/settings.json to pattern-match read-only Bash / git / gh operations past the classifier entirely.

Five distinct 400-error classes surface when running Claude Code against Bedrock-hosted OSS models (Qwen, Kimi, MiniMax, DeepSeek): the stopSequences field rejected by Bedrock Converse for several third-party families; the 64-character toolSpec.name limit; bare-string tool_result content that Bedrock auto-parses and rejects unless wrapped as a JSON object; max_tokens=32000 overflowing the upstream context window mid-session; and a missing POST /v1/messages/count_tokens handler that made the client blast ahead blind. The public adapter now handles all five, end-to-end-verified against qwen.qwen3-coder-480b-a35b-v1:0 on EPAM DIAL. The 2026-05-16 hardening finding walks each class with the per-layer test.

13 Scope limits · What this report does not claim

The contract between the report and the next reader who acts on it.

Each finding has a scope. Stating the scope as plainly as the finding is what keeps the report from being over-read into contexts where the magnitudes do not transfer.

  1. Single-engineer data. All 192 trials originate from one engineer’s laptop, one DIAL deployment, one network path to ai-proxy.lab.epam.com. Latency tails include network RTT variance that production deployments would see differently. The pattern (route Claude Code through DIAL, measure $/success under list pricing, instrument the wire) transfers across delivery contexts; the specific magnitudes do not transfer without re-measurement.
  2. Synthetic task surface. The 8-task portfolio is reproducible and discriminating but is not a stand-in for any specific real codebase. T8 (refactor + test + document) is the closest to a realistic refactor; T1 through T7 are unit-test shaped.
  3. Bedrock list pricing, not EPAM DIAL contract pricing. Every $/M rate in the cost table is taken from the AWS Bedrock pricing console for US East Ohio on 2026-05-14. EPAM DIAL contract pricing may include a markup or discount that shifts absolute magnitudes; the qualitative ranking inside the 100%-pass cluster is stable under markup factors below ~6.7× on Qwen 480B input rates.
  4. Part III is a description of latent capability, not a deployed system. The three tag promotions and header allowlist described in Section 12 are not in production. Illustrative numbers in Sections 06b, 07, and 09 are not measurements from the current data and are labelled as such.
  5. No code-quality cross-check. Hidden validators catch syntactic and functional correctness but not code quality. A model could pass T2 by producing semantically-correct-but-stylistically-poor code; the matrix would call this a pass.
  6. Iteration-2 substrate update is single-config. The Kimi K2.5 + Qwen 480B + MiniMax decoupling described in §12 was verified against one client config (Claude Code 2.1.142) over one representative session. Whether the namer / summariser cost lever holds at scale, and whether classifier-path hardcoding changes in subsequent Claude Code releases, are open questions.
  7. The sanitised public adapter is a subset, not the whole. The repo at github.com/agorokh/sdlc-dial-adapter carries the translation core, the public log schema, the Grafana dashboard, and the 52-test suite. It does not carry the price-table calibration, the Influx tenant presets, or the launcher tooling that ties everything to one machine. The reader who clones it gets a working adapter; they do not get the local sandbox.
14 Closing · The take-away

One gateway, one harness, two cost levers, one observation point.

The headline numbers in this report do not generalise; the pattern does. Routing a frontier coding harness through an enterprise gateway is one specific instance of the Applied AI bridging discipline. Two cost levers (the model choice inside the 100%-pass cluster, and the namer / summariser decoupling) fall out once the gateway is in the path. The structured observation point that makes both levers measurable is the same wire the adapter already translates.

The closing thesis

The gateway is the right boundary at which to enforce tenancy, audit, routing, and observability. A coding harness that runs through it gets all four for free.

Frontier labs ship harnesses with vendor-locked endpoints by default. Consumer products ship them with hidden upstream routing. The Applied AI discipline is the work that puts the harness on the gateway every other enterprise LLM call already goes through, then measures what changes. The sibling report in this series closes the same gap on a different capability: long-term memory for the agents the same harness runs.

A Methods appendix · For staff who would re-run this

The apparatus, the routing, the artefacts.

Everything in the body discussed what the numbers mean. This appendix documents what they are: harness mechanics, pricing methodology, telemetry schema, decoupled-model routing table, and the public artefacts a reader can clone.

Harness

Claude Code 2.1.142 invoked through a thin wrapper that sets ANTHROPIC_BASE_URL, the four model-slot env vars, and CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1 (DIAL gateways 400 on unknown beta headers). The 8-task portfolio fires each task three times per upstream (24 trials per upstream, 192 trials across the eight upstreams). Hidden pytest validators score each trial as pass / fail; no LLM-as-judge is used in this report (the matrix is functional correctness, not preference).

Pricing methodology

Per-trial cost is computed from the per-request structured JSON log line (input tokens + output tokens + cache-read tokens, when the upstream supports cache_control) multiplied by the AWS Bedrock pricing-console rates for US East Ohio on 2026-05-14. EPAM DIAL contract pricing is out of scope (see §13).
