00 Cover · the middle band

Claude Code via EPAM DIAL · POC report.

Frontier labs ship coding harnesses pointed at the vendor's own endpoint; consumer products lock the same harness to a single upstream; enterprise systems still have to survive the governance contract every regulated delivery context insists on. Applied AI is the middle band — the discipline of taking a research result or a consumer-grade feature and converting it into a substrate with measured properties, named failure modes, and an adoption path. This entry closes the gap on one capability: routing Anthropic's Claude Code 2.1 through EPAM's DIAL gateway, measuring eight upstreams under one fixture, and treating the adapter as a structured observation point for the AI-native SDLC.

01 Executive summary

Three findings, one strategic angle.

Structured as a deployment-decision memo, not a vendor benchmark recap. Three callouts are the headline; the body sections that follow are the supporting argument.

Finding 1 — The bridge works on the EPAM-compliant inference path

Claude Code 2.1 talks to any DIAL-hosted upstream through one Anthropic-to-OpenAI translation adapter. 192 trials produced zero authentication incidents and required no Claude Code source modifications. Traffic stays inside the EPAM perimeter the gateway already controls.

Finding 2 — Qwen3-Coder 480B is the cost leader at $0.101 per success

At Bedrock list pricing, Qwen 480B is the cheapest 100%-pass model in the matrix — 38% below Claude Haiku 4.5, 4.7× below Sonnet 4.6, 6.8× below Opus 4.7. Three OSS upstreams below the Sonnet tier fail with a scale-driven, vendor-independent signature.

Finding 3 — Per-request telemetry, wired today plus a clean follow-up

Each request writes a structured record with tokens, tool inventory, cache behaviour, latency, and Bedrock-priced cost. Two business dimensions (client, model family) are wired today. Three more identifiers already in the wire can become tags with no client change. Business-context tags — project, team, industry, use-case — remain a follow-up.

02 Why bridging matters

Frontier harnesses ship pointed at one endpoint. Enterprises run on every endpoint.

Single-vendor coding harnesses are the 2026 default: Claude Code points at Anthropic, Codex CLI at OpenAI, Cursor at its own router. Each is excellent at its first-party model and progressively worse as the user swaps in a foreign upstream. The failures that matter for enterprise adoption appear when the same harness has to run on whichever inference path the procurement, governance, and FinOps stacks already sanction.

That gateway is rarely the vendor's own. In EPAM's case it is DIAL — a project-keyed, audit-logged proxy that already mediates every other LLM call inside the perimeter. The question this report tests is whether a frontier coding harness can route through that gateway without losing the harness's quality, and what business-grade telemetry the gateway exposes in exchange.

What "bridging" means here

Three structural choices, in combination, define an enterprise-routed harness:

  • Where the requests originate. Engineer laptop, CI runner, IDE plugin, hosted notebook.
  • How they reach an upstream. Direct vendor endpoint, single-tenant gateway, multi-tenant gateway with project routing.
  • What gets retained on the wire. Plain transit, request/response logging, full telemetry with business dimensions.

The POC here is laptop → DIAL → Bedrock-backed upstream, with full per-request telemetry retained at the adapter.

06 Reliability — telemetry retained on the wire

The adapter sees every request. That is the strategic angle.

Each request through the adapter writes a structured JSON record to InfluxDB measurements. The wire already carries enough to reconstruct what the harness asked, what the upstream returned, what it cost, and how long it took.

What gets retained today

~30 fields per request: input/output token counts, tool inventory hash, cache-read/cache-write tokens, latency, Bedrock-priced cost, client name, target model family. Three more in-flight identifiers (session id, tool inventory hash, MCP servers seen) can be promoted to tags with no client-side change. Business-context tags (project, team, industry, use-case) remain a follow-up subject to data-governance policy.

The reliability claim here is not "the adapter never drops a request" (the POC sample is too small for that). It is that a centralised version of the same adapter inside DIAL would expose, per request, what every other LLM call inside the perimeter already exposes — and add the coding-agent dimensions on top.

09 Platform implications

What the gateway has to expose.

If the bridge pattern is the thing that wins, the gateway contract that wraps it needs four properties beyond what the POC adapter exercises today.

Per-request telemetry as a first-class side-channel

The ~30 fields the POC adapter already retains belong on the gateway's own observability path, not on a per-laptop sidecar. The centralised version is what makes the wire a data plane.

Business-context tags joinable to the wire

Project, team, industry, use-case dimensions joined at the gateway, supplied by the existing project metadata and authentication. The POC writes client and model-family tags today; production joins all four business-context dimensions on the same path.

Per-project routing rules at the gateway

Model swap, fallback chain, rate-limit, and audit policy expressed as gateway configuration — not client environment variables. The bridge is the right place to enforce them.

Decoupled cost accounting per upstream

Bedrock-priced cost per call, attributed back to the project and team that triggered it. Cost-per-success becomes a per-team metric, not a fleet average.

The DIAL ask

The POC laptop-adapter pattern proves the bridge works and exposes what the wire could carry. The next contract increment is moving the translation adapter into DIAL itself — same shape, same behaviour, gateway-side telemetry, gateway-side routing rules. The sibling report (Choosing memory for enterprise agents) covers the substrate-selection question that runs on top of the same DIAL routing.

10 Recommendations

Three for engineering, two for the platform.

  1. Standardise on the bridge. Every Claude Code 2.1 deployment inside EPAM routes through DIAL by end of Q3 2026. Direct vendor-endpoint calls become an exception, not a default.
  2. Default upstream is Qwen 480B. The cost-leader of the tier-A cluster at $0.101 per success. Anthropic upstreams remain available on opt-in for tasks scoped to need them.
  3. Decouple namer / summariser from the main loop. The Claude Code 2.1 design has three model paths; the cheaper summariser path (Kimi K2.5) reroutes via ANTHROPIC_SMALL_FAST_MODEL with no main-loop quality impact.
  4. Translation adapter inside DIAL. The same POST /v1/messages → OpenAI chat-completions translation that the POC runs per-laptop becomes a gateway-side surface. Removes the client-side env-var ceremony.
  5. Per-request telemetry as a first-class gateway artefact. The ~30 fields the adapter retains today move to the gateway's observability path with business-context tags joined at write.

11 Scope limits

What this report does not cover.

  • OSS harness coverage. Cursor, Aider, Codex CLI, OpenHands, Continue, and Cline sit outside this evaluation. The pattern (route through gateway, measure $/success, instrument the wire) transfers; the magnitudes do not transfer without re-measurement on each harness.
  • Non-coding workloads. The eight tasks are repository-shaped. A customer-support agent or a regulatory-research agent has different cost-quality pressure on the same gateway pattern.
  • Long-running sessions. The longest session in this matrix is one engineering day. Multi-day session behaviour belongs to the sibling memory-substrate report, not this one.
  • EPAM contract pricing. All cost figures use AWS Bedrock list pricing. EPAM's negotiated rates are out of scope for this document.

12 Closing

Applied AI is the work of moving the gateway to the right place.

The headline number across this engagement is not the $0.101 per success. It is that we know the number because the bar was stated before the trials ran and the bridge stayed inside the perimeter the gateway already controls. Restate the bar, retest the upstreams, republish the number — that is the loop. This entry is one rotation on one capability. The sibling report closes a different rotation on the same DIAL routing.
